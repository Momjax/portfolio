RP1 - Classcord

Application de messagerie instantanée en Java inspirée de Discord

Terminé Formation (École) Java Swing TCP/JSON

1. Contexte

Situation

Ce projet a été réalisé dans le cadre du BTS SIO SLAM lors d'une semaine intensive de développement Java. L'objectif était de créer une application de messagerie fonctionnelle démontrant nos compétences en programmation réseau et interface graphique.

Organisation

Projet développé en binôme avec un camarade SISR. J'ai principalement travaillé sur le développement client et l'interface graphique, en collaboration sur le protocole de communication.

Problématique

Comment concevoir une application de messagerie temps réel permettant à plusieurs utilisateurs de communiquer via une architecture client-serveur fiable ?

2. Démarche d'Analyse

A. Analyse des besoins

  • Créer une messagerie instantanée temps réel (type Discord)
  • Gérer plusieurs connexions simultanées, le statut des usagers et les MP

B. Pistes et hypothèses envisagées

  • Réseau : Sockets UDP (rapide mais perte possible) vs TCP (fiable, priorisé).
  • Format de données : XML (trop lourd) vs Sérialisation Java (couplé) vs JSON (léger et universel).
  • Architecture code : Tout regrouper vs Séparer via MVC (Modèle-Vue-Contrôleur).

C. Solution retenue & Justification

Développement Java/Swing (imposé) avec une architecture Client-Serveur TCP pour la fiabilité. Le format JSON a été choisi pour standardiser les messages, et le pattern MVC appliqué sur le client (Vue Swing / Thread Récepteur / Contrôleur Métier) afin d'assurer l'évolutivité du code.

3. Solution Proposée

Architecture client-serveur

L'application Classcord utilise une architecture classique :

  • Serveur : Gère les connexions, route les messages, maintient l'état
  • Client : Interface Swing pour l'interaction utilisateur
  • Protocole : Messages JSON via sockets TCP

5. Cadrage et Gestion de Projet

Équipe et Rôles

  • Moi : Référent développement, conception logicielle, architecture réseau TCP et interface graphique Swing.
  • Binôme : Appui technique sur les tests et l'infrastructure de lancement du serveur.

Parties Prenantes

  • Commanditaire : Professeurs encadrants (valident l'avancement et le code source).
  • Utilisateurs cibles : Faux utilisateurs pour benchmarker les salons de chat.

Démarche & Besoins

Cahier des charges initial strict fourni par l'équipe pédagogique. Les besoins ont été découpés en "User Stories" techniques et priorisés. Le produit devait être intégralement codé en Java.

Contraintes & Risques

  • Risque technique : Crash serveur si surcharge de connexions (Mitigation: utilisation de threads isolés).
  • Contraintes : Temps de réalisation court (quelques semaines).

Planification et Suivi de Projet

6. Difficultés & Solutions

Problème 1 : Synchronisation des threads

L'interface se figeait lors de la réception de messages.

Solution :

Utilisation de SwingUtilities.invokeLater() pour les mises à jour UI depuis le thread réseau.

Problème 2 : Parsing JSON instable

Erreurs lors de messages mal formatés.

Solution :

Try-catch systématique et validation du JSON avant traitement.
