Veille Technologique

Problématique : Comment REST et GraphQL évoluent-ils en 2024-2025 pour répondre aux besoins croissants de sécurité, de performance et de flexibilité dans le développement d'APIs modernes ?

Introduction

Dans le cadre de ma formation BTS SIO SLAM, j'ai mené une veille technologique sur les APIs REST et GraphQL. Ces technologies sont essentielles pour permettre la communication entre applications, services web et bases de données.

Ma veille porte sur l'évolution récente de ces deux approches : REST, l'architecture traditionnelle et éprouvée, et GraphQL, l'alternative moderne qui gagne en popularité. J'analyse également les enjeux de sécurité avec JWT et OAuth 2.0, qui sont devenus incontournables en 2024-2025.

REST API

L'architecture standard pour les APIs web

Qu'est-ce que REST ?

REST (Representational State Transfer) est un style d'architecture pour concevoir des APIs. Il utilise les méthodes HTTP standards (GET, POST, PUT, DELETE) et des URLs pour accéder aux ressources.

REST fonctionne avec des endpoints multiples : chaque ressource (utilisateur, produit, commande) possède sa propre URL. Par exemple :

GET /api/users/123
GET /api/users/123/orders
POST /api/orders

Forces et limites en 2025

✓ Points forts :

  • Simple à comprendre et implémenter
  • Caching HTTP très efficace
  • Outils et écosystème matures (Postman, Swagger)
  • 83% des APIs web utilisent REST en 2024

✗ Limites :

  • Over-fetching : récupère trop de données
  • Under-fetching : nécessite plusieurs requêtes
  • Versionnement complexe (v1, v2...)

GraphQL

La nouvelle génération d'APIs

Qu'est-ce que GraphQL ?

GraphQL est un langage de requête pour APIs développé par Facebook en 2012 et rendu open-source en 2015. Contrairement à REST, GraphQL utilise un seul endpoint et permet au client de demander exactement les données dont il a besoin.

Exemple de requête GraphQL :

query {
  user(id: "123") {
    name
    email
    orders {
      total
    }
  }
}

Une seule requête récupère toutes les données liées !

Adoption massive en 2024-2025

✓ Avantages :

  • Flexibilité totale : le client choisit les données
  • Une seule requête au lieu de plusieurs
  • Typage fort avec schéma auto-documenté
  • Support temps réel (Subscriptions)
  • Adoption +340% chez Fortune 500

⚠ Défis :

  • Courbe d'apprentissage plus élevée
  • Caching plus complexe
  • Requêtes très complexes possibles

Tendances 2024-2025

61% des entreprises utilisent GraphQL en production (Hygraph 2024)

10% ont complètement remplacé REST par GraphQL

89% des équipes qui adoptent GraphQL le choisiraient à nouveau (Apollo Survey 2024)

Utilisé par : Facebook, GitHub, Shopify, Airbnb

REST vs GraphQL : Lequel choisir ?

Analyse comparative pour 2025

Critère REST GraphQL
Endpoints Multiples (un par ressource) Un seul endpoint
Requêtes de données Fixes (over/under fetching) Exactement ce qui est demandé
Performance Excellente pour CRUD simple Meilleure pour données complexes
Caching ✓ Très simple (HTTP) ⚠ Plus complexe
Documentation Manuelle (Swagger) Auto-générée (schéma)

Conclusion : REST et GraphQL coexistent en 2025. Le choix dépend du contexte : REST pour sa simplicité et GraphQL pour sa flexibilité.

Sécurité des APIs

JWT et OAuth 2.0 : Les standards 2024-2025

JWT (JSON Web Token)

Le JWT est un standard (RFC 7519) pour transmettre des informations d'authentification de manière sécurisée. Un JWT est composé de 3 parties :

  • Header : Type et algorithme (HMAC-SHA256, RSA)
  • Payload : Données utilisateur (ID, rôle, expiration)
  • Signature : Garantit l'intégrité du token

eyJhbGci.eyJzdWIi.SflKxwRJ

Avantages :

  • Stateless (pas de stockage serveur)
  • Compact et rapide
  • Multi-plateforme

OAuth 2.0

OAuth 2.0 est un framework d'autorisation qui permet à une application d'accéder à des ressources protégées au nom d'un utilisateur, sans partager ses identifiants.

OAuth 2.0 est utilisé par Google, Facebook, GitHub pour la connexion tierce ("Se connecter avec Google").

Fonctionnement :

  • L'utilisateur se connecte au fournisseur (Google)
  • Le serveur d'autorisation génère un access_token
  • L'app utilise ce token pour accéder aux ressources
  • Le token expire après un certain temps (sécurité)

Souvent combiné avec JWT pour transporter les tokens d'accès.

Bonnes pratiques de sécurité 2025

À faire :

  • Toujours utiliser HTTPS (TLS)
  • Limiter la durée de vie des tokens (TTL)
  • Utiliser des clés complexes et aléatoires
  • Vérifier la signature JWT
  • Implémenter un système de bannissement

À éviter :

  • Stocker JWT dans localStorage (XSS)
  • Tokens trop longs (>1h sans refresh)
  • Algorithme "none" dans JWT
  • Secrets dans le code source
  • Négliger la validation des entrées

Conclusion

Ma veille montre que REST et GraphQL ne sont pas des concurrents, mais des solutions complémentaires qui coexistent en 2025.

REST reste le standard pour les APIs publiques, les opérations CRUD simples et les architectures microservices. Avec 83% de parts de marché, REST n'est pas prêt de disparaître grâce à sa simplicité et son écosystème mature.

GraphQL s'impose dans les applications complexes, notamment mobiles, où la flexibilité et la réduction des requêtes sont cruciales. Son adoption a bondi de 340% chez les grandes entreprises, et 61% des organisations l'utilisent maintenant en production.

Concernant la sécurité, JWT et OAuth 2.0 sont devenus incontournables pour protéger les APIs. Les bonnes pratiques 2025 incluent HTTPS obligatoire, tokens à durée de vie limitée, et protection contre les attaques XSS et CSRF.

Ce que cette veille m'a apporté :

J'ai compris que le choix d'une architecture API dépend du contexte projet : complexité des données, type de client (web/mobile), et besoins de performance. J'ai également appris l'importance de la sécurité dès la conception, avec JWT et OAuth 2.0 comme standards modernes. Ces connaissances me seront essentielles pour mes futurs projets en tant que développeur.
