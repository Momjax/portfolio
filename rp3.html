RP3 - Jeu d'Arène

Jeu multijoueur en Java avec architecture client-serveur

Terminé Java Swing Multithreading

1. Contexte

Situation

Ce projet a été réalisé dans le cadre du BTS SIO SLAM comme projet de programmation Java avancée. L'objectif était de créer un jeu multijoueur démontrant nos compétences en programmation réseau, interface graphique et gestion de la concurrence.

Organisation

Projet individuel développé sur 6 semaines avec des points d'étape réguliers pour valider l'avancement des différents modules.

Problématique

Comment concevoir un jeu multijoueur temps réel avec une synchronisation efficace entre plusieurs clients et un serveur, tout en gérant les contraintes de performance et de latence réseau ?

2. Besoin & Objectifs

Objectifs fonctionnels

  • Connexion simultanée de plusieurs joueurs
  • Déplacement des personnages dans une arène 2D
  • Génération aléatoire d'obstacles (murs)
  • Chat intégré pour communiquer entre joueurs
  • Synchronisation en temps réel des positions

Contraintes techniques

  • Architecture client-serveur obligatoire
  • Java + Swing pour l'interface
  • Gestion multi-thread côté serveur
  • Fluidité d'affichage (minimum 30 FPS)

Extrait de code - Gestion des collisions

public boolean checkCollision(Player player, int newX, int newY) {
    Rectangle playerBounds = new Rectangle(newX, newY, 
                                            PLAYER_SIZE, PLAYER_SIZE);
    
    for (Wall wall : walls) {
        if (playerBounds.intersects(wall.getBounds())) {
            return true; // Collision détectée
        }
    }
    
    for (Player other : players) {
        if (other != player && 
            playerBounds.intersects(other.getBounds())) {
            return true;
        }
    }
    
    return false;
}

6. Difficultés & Solutions

Problème 1 : Race conditions

Accès concurrent aux données du jeu causant des incohérences.

Solution :

Utilisation de synchronized et ConcurrentHashMap pour les structures partagées.

Problème 2 : Latence réseau

Décalage visible entre les mouvements locaux et distants.

Solution :

Interpolation côté client et prédiction de mouvement pour fluidifier l'affichage.

Problème 3 : Fuite mémoire

Les threads clients non fermés proprement lors des déconnexions.

Solution :

Implémentation d'un mécanisme de cleanup avec try-finally et fermeture explicite des sockets.
