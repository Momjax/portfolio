RP5 - ObsiLock

API REST de gestion de fichiers sécurisés - Coffre-fort numérique

Terminé Formation (École) PHP Slim Framework Medoo MySQL Docker

1. Contexte

Situation

Ce projet a été réalisé dans le cadre d'un TP sur les API REST et l'architecture MVC en 2ème année de BTS SIO SLAM. L'objectif était de créer une API REST complète permettant de gérer un coffre-fort numérique de fichiers avec upload, téléchargement, et gestion de quota.

Organisation

Projet individuel développé sur 4 semaines en suivant un cahier des charges précis. Utilisation de Slim Framework (PHP) et Medoo (micro-ORM) pour une architecture MVC propre.

Problématique

Comment concevoir une API REST RESTful permettant de gérer des fichiers de manière sécurisée, avec gestion de quotas, tout en respectant les bonnes pratiques d'architecture logicielle ?

2. Démarche d'Analyse

A. Analyse des besoins

  • Coder une interface pour uploader, lister et télécharger des fichiers.
  • Gérer des quotas d'utilisations.
  • Mettre à disposition une interface sous forme d'API REST exploitable par un client lourd ou mobile.

B. Pistes et hypothèses envisagées

  • Type de Backend : Serveur monolithique "old-school" vs API REST Stateless.
  • Technologies PHP : Framework complet (Symfony/Laravel) vs Micro-framework (Slim).
  • Interopérabilité : Format XML vs JSON (standard API) + Authentification JWT.

C. Solution retenue & Justification

Pour plus de légèreté et répondre uniquement au besoin d'API, Slim Framework a été sélectionné, couplé à Medoo comme micro-ORM. L'application est décorrélée du front-end grâce au principe de l'API REST et produit uniquement des réponses JSON. Pour garantir un environnement identique du développement à la production, l'ensemble est conteneurisé avec Docker.

Concept Technique : JSON Web Token (JWT)

Qu'est-ce qu'un JWT ? C'est un standard ouvert (RFC 7519) permettant d'échanger des informations sécurisées entre un client et un serveur sous forme d'objet JSON signé crytographiquement.

Pourquoi ce choix ? Les API REST doivent être sans état (stateless). JWT permet d'authentifier un utilisateur sans que le serveur n'ait à stocker de session. La validité est assurée par la signature du token.

Usage dans ObsiLock : À la connexion, l'API génère un JWT valide pendant une durée déterminée. Le client HTTP renvoie ce JWT dans le header `Authorization: Bearer ` de chaque requête pour prouver son identité et accéder à ses fichiers liés.

class FileRepository







{







    private Medoo $db;







    public function __construct(Medoo $db)







    {







        $this->db = $db;







    }







    public function listFiles(): array







    {







        return $this->db->select('files', '*');







    }







    public function create(array $data): int







    {







        $this->db->insert('files', $data);







        return (int)$this->db->id();







    }







    public function totalSize(): int







    {







        return (int)$this->db->sum('files', 'size') ?: 0;







    }







}

5. Cadrage et Gestion de Projet

Équipe et Rôles

  • Moi : Contexte de développement fullstack / API. Conception du back-end de A à Z.
  • Binôme : Déploiement et tests automatisés. (Délégation d'une partie des endpoints REST).

Parties Prenantes

  • Commanditaire : Validation profils enseignants (Bloc 2 / API).
  • Utilisateurs "Machine" : Les clients lourds/front-end qui consommeront cette API.

Démarche & Besoins

Le projet a démarré sur la base d'une spécification d'API stricte (endpoints attendus, codes retour HTTP 200, 401, 404). L'implémentation de la norme JWT ajoutait une contrainte de sécurité essentielle nécessitant des revues récurrentes.

Contraintes & Risques

  • Risque sécurité : Contournement du quota d'upload par modification de requêtes.
  • Contraintes : API obligatoirement RESTful & architecture Docker.

Planification et Suivi de Projet

La gestion s'est orchestrée en méthode Agile. Les différentes routes de l'API (GET, POST, DELETE) ont été transformées en User Stories consignées dans un Kanban partagé. Les livraisons (Sprints) permettaient de tester indépendamment chaque endpoint via des outils comme Postman.

6. Difficultés & Solutions

Problème 1 : Routing Slim avec sous-dossier

L'API ne répondait pas correctement quand servie depuis un sous-dossier.

Solution :

Utilisation de setBasePath() pour auto-détecter le chemin de base.

Problème 2 : Gestion du quota

Éviter le dépassement lors d'uploads simultanés.

Solution :

Vérification du quota avant l'upload avec calcul de la taille totale.
